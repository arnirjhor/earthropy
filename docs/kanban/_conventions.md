# Kanban conventions

Source of truth for what's next is `docs/kanban/tasks/*.md`. `INDEX.md` is a derived view (regenerated by the Orchestrator). When the GitHub remote arrives, these files mirror to GitHub Issues with the same IDs.

## Task file shape

Filename: `<id>.md`. ID format: `<PHASE>-<AREA>-<ORDINAL>` (e.g., `A-AUTH-1`, `B-GROUP-3`, `X-VITEST-1`). `X` is for cross-cutting infrastructure.

Frontmatter is YAML; body is markdown. Required fields:

```yaml
---
id: A-AUTH-1
title: "<short imperative title>"
status: backlog | ready | in_progress | in_review | done | blocked | cancelled
priority: critical | high | medium | low
phase: A | B | C | D | E | F | X
agent_model: opus | sonnet | haiku
deps: []                                  # list of task ids; empty if none
tags: [auth, security]
owner: ""                                 # set by Orchestrator on dispatch
branch: ""                                # set by Builder
pr: ""                                    # PR URL or "local:<branch>" while no remote
estimated_hours: 4
created: 2026-05-18
updated: 2026-05-18
---

## Description
What needs to exist when this task is done. One short paragraph.

## Acceptance criteria
- [ ] Each item is independently testable. Vague items are bugs.

## Test plan
List the Vitest / Playwright tests the Builder must write *before* the implementation.

## Notes
Updated during work — Builder appends decisions, Reviewer appends comments, Orchestrator appends dispatch trace.
```

## Status lifecycle

```
backlog ──(all deps done)──▶ ready ──(agent claimed)──▶ in_progress
                                                          │
                                                          ▼
                                                       in_review
                                                          │
                                                          ▼
                                                         done
        ▼
     blocked   (with reason in Notes; deps may need attention)
        ▼
     cancelled (with reason in Notes; out of scope)
```

A task moves `ready` only when every `deps` task is `done`. The Orchestrator scans `tasks/*.md` each wake cycle and promotes `backlog → ready` accordingly.

## Priority

- `critical`: blocks v0.1 ship; on the critical path.
- `high`: in scope for v0.1; not on critical path.
- `medium`: in scope; can slip a sub-phase.
- `low`: nice-to-have; defer if budget tight.

## Agent model

`opus` for architecture + hard debugging + creative design. `sonnet` for Builders + Reviewers + Researchers (default). `haiku` for Testers, Documenters, Translators. Set this when the task is created; the Orchestrator honors it on dispatch.

## Dependencies

`deps` lists task IDs that must reach `done` before this task is `ready`. Be precise — a single missed dep wakes Reviewers to red flags.

## INDEX.md

A flat, grouped, derived view. The Orchestrator regenerates it after each batch. Do not hand-edit `INDEX.md`.

## One-time setup

After `pnpm install`, install the Playwright browser binary: `pnpm exec playwright install --with-deps chromium`
